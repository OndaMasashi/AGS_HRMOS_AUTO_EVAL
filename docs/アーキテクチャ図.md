# HRMOS 応募者 AI 自動評価パイプライン — アーキテクチャ図

インタラクティブ版: [アーキテクチャ図.html](アーキテクチャ図.html)（ブラウザで開く。フローを選んで ▶ Play すると1ステップずつ流れを追える）

HRMOS採用ページから応募者書類（PDF/Word/Excel）を自動取得し、PIIマスキング（個人情報の伏字化）のうえ Claude / Gemini CLI で評価・スコアリングし、Excelレポート出力とメール通知まで行う5層パイプラインの可視化。

- **モード切替（右上トグル）**: 評価プロバイダ `claude` / `gemini`（`config.yaml` の `evaluation.provider`）。LLM CLI ノードのラベル・呼び出しコマンドが切り替わる。
- **ショートカット**: `Space` 再生/停止 · `←`/`→` ステップ · `1`–`4` フロー · `O` プロバイダ切替 · `T` テーマ · `F` 全画面 · `R` 配置リセット。ノードはドラッグで再配置可。

## コンポーネント（ノード）

| ノード | 役割 | 実装 |
| :--- | :--- | :--- |
| **run.py / main.py** | CLIエントリ＆オーケストレーター（司令塔）。各層を順次呼び出す | `run.py`（argparse）→ `src/main.py: run_scan/run_report/show_status` |
| **HRMOS 採用ページ** | 外部Web。応募者一覧・添付書類の取得元（認証必須） | — |
| **ブラウザ操作** | Playwright で HRMOS ログイン・一覧巡回・添付DL | `src/browser/`（auth・navigator・selectors） |
| **パーサ** | PDF/DOCX/XLSX → テキスト抽出（80,000文字上限） | `src/parser/document.py`（pdfplumber・python-docx・openpyxl） |
| **PII マスカー** | LLM送信前に氏名/電話/住所を伏字化し、応答後に復元 | `src/evaluator/pii_masker.py` |
| **LLM CLI** | `subprocess` で Claude/Gemini CLI を起動し評価JSONを取得（APIキー不要） | `src/evaluator/`（llm_client・claude_client・gemini_client） |
| **SQLite DB** | 応募者・書類・評価・実行履歴を保存（Repositoryパターン） | `src/database/`（models・repository）/ `data/hrmos.db` |
| **Excel レポータ** | レーダーチャート・S〜Dランク色付きの評価表を生成 | `src/reporter/export.py`（openpyxl）/ `data/reports/*.xlsx` |
| **メール通知** | Resend で評価サマリ・新規0件通知・失敗アラートを送信 | `src/reporter/notify.py` |

> 凡例の色は機能グループに対応: CLI/司令塔（緑）・ブラウザ（青）・パース/マスク（黄）・LLM評価（赤）・DB/出力（紫）・外部HRMOS/Resend（橙）。

## フロー

### 1. フルスキャン（`python run.py scan`）— 10ステップ

書類DL → AI評価 → Excel出力 → メール通知の全工程。`main.py: run_scan()` が司令塔。

1. **ブラウザ起動・セッション復元** — config読込・DB初期化・scan_run作成 → Playwright で chromium 起動、`storage_state.json` があればセッション復元。
2. **HRMOS ログイン & 一覧巡回** — `ensure_authenticated`。失敗時は scan_run を fail にして失敗アラートを送り中断。成功後に応募者一覧リンクを収集。
3. **応募者リンク・添付を取得** — DBへ upsert し対象を絞る（既定 `pending` / `--all` 全件 / `--retry-errors`）。各ページから添付リンク取得。
4. **添付DL → テキスト抽出** — `data/downloads/<app_id>/` にDL、形式別にテキスト化、documents に記録、全書類を結合（80k文字上限）。
5. **PII マスキング（送信前）** — 氏名/電話/住所を `[NAME_1]` 等へ置換し評価プロンプトを構築（失敗時はマスクなしで続行・警告）。
6. **LLM CLI で評価** — provider に応じて `claude -p` / `gemini` を `subprocess` 起動（stdin→stdout JSON、最大3回リトライ、timeout 300秒）。Claude時は `CLAUDECODE` 環境変数を除去しネストを防止。
7. **JSON 応答 → アンマスク** — JSONパース＆criteria検証後、コメント・総合評価・面接質問・備考のプレースホルダーを元PIIへ復元。
8. **評価結果を DB 保存** — 1応募者×評価基準数の行を一括INSERT。status を `scanned`（失敗時 `error`）へ。
9. **run 集計 → Excel 出力** — `eval_count>0` なら今回runの評価でExcel生成（基本情報→1次通過○/△→平均/合計→S〜Dランク→レーダーチャート→各基準→備考→質問候補）。
10. **メール通知（○の経歴書添付）** — Resendでサマリ＋Excel送信。`attach_resumes` が true なら1次通過候補(○)の経歴書を添付（重複パス排除）。※添付はマスクなしPIIを含むため運用注意。

### 2. PII マスキング往復 — 4ステップ

個人情報をLLMに渡さず、応答で復元する仕組みにフォーカス。`parser → masker → llm → masker → db` のみが点灯し、他ノードは暗転する。

1. 結合テキスト受領・PII検出（マッピング表作成）
2. マスク済みテキストだけを LLM 送信（**生PIIはプロセス外＝Claude/Gemini に一切出ない**）
3. プレースホルダー入りのまま応答JSONを受領
4. `_unmask_evaluation_data` で元PIIへ復元しDB保存

### 3. Excel 再出力（`python run.py report`）— 3ステップ

DB内の既存評価を再集計してExcelのみ出力（HRMOS/LLMにアクセスせず、**再評価しない**）。

1. `--run-id` 指定で該当run、無指定で全評価をDBクエリ（0件なら終了）
2. scanと同じ `export.py` でExcel生成
3. パスと評価者数を標準出力に表示。**メール送信は行わない**（scanとの違い）

### 4. メール通知の分岐 — 3パターン（`notify.py`）

scan終了時、結果に応じて送り分ける。

- **A. 成功通知**（`eval_count>0`）: 評価サマリ・内訳・○のHRMOSリンク・Excel・○候補の経歴書添付。
- **B. 新規0件通知**（`targets` 空で正常終了）: `notify_on_no_candidates=true` なら「新規なし」を通知し、無音による誤認を防ぐ。**0件評価は失敗ではない**。
- **C. 失敗アラート**（常時送信）: ①認証失敗 ②応募者一覧0件 ③評価対象はいたが成功0件（全員エラー/添付なし）④致命的例外。「新規なし」と誤報して障害を隠さないための分岐。

## モード差分（claude / gemini）

| 項目 | claude | gemini |
| :--- | :--- | :--- |
| LLM CLI ラベル | Claude CLI | Gemini CLI |
| 呼び出し | `subprocess.run(["claude","-p"], …)`、`CLAUDECODE` 除去 | `subprocess.run(["gemini", …], …)` |
| 共通 | stdin→stdout JSON・最大3回リトライ・timeout 300秒・APIキー不要 | 同左 |

トグルで切り替わるのは LLM CLI ノードのみ。パイプラインの形（ノード構成・接続）はプロバイダに依存しない。

## ワークショップ／レビューでの使い方

- **新規メンバーのオンボーディング**: フロー1を Play で流し、5層がどう連携するかを通しで説明。
- **PII保護の説明**: フロー2でノードが暗転する様子を見せ、「生の個人情報は Claude/Gemini に届かない」ことを視覚的に保証。
- **障害対応の合意形成**: フロー4で「どの異常がどのメールになるか」を共有し、無音＝正常 vs 失敗アラートの判断基準を揃える。
- **プロバイダ選定**: 右上トグルで Claude/Gemini の呼び出し差分を比較。
